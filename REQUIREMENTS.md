# Техническое задание v1.1: RaceNotes — MVP

## 1. Общая информация о проекте

### 1.1. Название проекта
**RaceNotes** — дневник велогонщика для учёта сетапов, планирования гонок и анализа результатов.

### 1.2. Заказчик
Даниил Киселев (велогонщик, технический директор)

### 1.3. Цель проекта
Создать PWA-приложение для велогонщиков, позволяющее вести учёт сетапов велосипеда, планировать гонки и анализировать результаты с акцентом на **давление в шинах**, **погодные условия** и **план питания**.

### 1.4. Целевая аудитория
- Активные велогонщики (10+ гонок в год)
- Полупрофессиональные спортсмены
- Велоклубы и команды

### 1.5. Ключевая ценность продукта
Быстро вспомнить какой сетап, давление в шинах и план питания использовал на прошлых гонках, чтобы повторить успешную конфигурацию.

### 1.6. Ключевые отличия от конкурентов
- Связь сетапа и гонки с результатами
- Калькулятор давления в шинах
- Учёт погоды и плана питания
- PWA с оффлайн режимом

## 2. Технологический стек

### Backend:
- Язык: Go 1.21+
- Фреймворк: Gin
- База данных: PostgreSQL 15+
- Авторизация: JWT
- ORM: GORM
- Email: SMTP или SendGrid

### Frontend:
- Фреймворк: Vue 3 (Composition API)
- CSS: Tailwind CSS
- Роутинг: Vue Router 4
- HTTP: Axios
- State: Pinia
- PWA: Vite PWA Plugin

### Инфраструктура:
- Контейнеризация: Docker, docker-compose
- Хостинг: VPS (арендованный сервер)
- Reverse proxy: Nginx
- SSL: Let's Encrypt
- Хранилище фото: Локально на сервере (Docker volume `uploads`)
- БД: PostgreSQL в контейнере с healthcheck, persistent volume `pgdata`

## 4. Модели данных

### 4.1. User (Пользователь)
```
User {
  id: UUID
  name: string (max 100)
  username: string (unique, max 50)
  email: string (unique, max 100)
  password: string (hashed)
  height: integer (см)
  weight: float (кг)
  created_at: timestamp
  updated_at: timestamp
  deleted_at: timestamp (nullable)
}
```

### 4.2. Setup (Сетап)
```
Setup {
  id: UUID
  user_id: UUID (FK -> User)
  name: string (max 100)
  photo: string (путь к файлу, nullable, загрузка через /api/upload)
  bike_name: string (max 100)
  tires: string (max 150)
  components_description: text (nullable)
  created_at: timestamp
  updated_at: timestamp
  deleted_at: timestamp (nullable)
}
```

**Бизнес-правило:** Давление НЕ хранится в Setup, только в Race (зависит от погоды).

### 4.3. Race (Гонка)
```
Race {
  id: UUID
  user_id: UUID (FK -> User)
  name: string (max 150)
  date: timestamp
  type: enum (Road, MTB, Gravel, Cyclocross, Track)
  photo: string (путь к файлу, nullable, загрузка через /api/upload)
  
  // Сетап
  setup_id: UUID (FK -> Setup, nullable)
  bike_name: string (nullable, если setup_id = null)
  tires: string (nullable, если setup_id = null)
  tire_pressure_front: float (bar, nullable)
  tire_pressure_rear: float (bar, nullable)
  other_components: text (nullable)
  
  // Погода
  temperature: integer (°C, nullable)
  conditions: enum (Sunny, Cloudy, Rain, Snow, nullable)
  wind: enum (None, Light, Moderate, Strong, nullable)
  road_conditions: enum (Dry, Wet, Mud, nullable)
  
  // План питания
  nutrition_plan: text (nullable)
  
  // Результат
  result: string (max 100, nullable)
  rating: integer (1-5, nullable)
  feelings: text (nullable)
  
  // Статус
  is_completed: boolean (default false)
  
  created_at: timestamp
  updated_at: timestamp
  deleted_at: timestamp (nullable)
}
```

**Бизнес-правила:**
- Если `setup_id != null`: bike_name и tires берутся из Setup
- Если `setup_id = null`: bike_name и tires обязательны
- Давление всегда указывается в Race (даже с готовым сетапом)
- `is_completed = false`: гонка запланирована
- `is_completed = true`: гонка прошла, результат заполнен

## 5. API Endpoints

### 5.1. Авторизация

**POST /api/auth/register** — Регистрация  
**POST /api/auth/login** — Авторизация  
**GET /api/user/profile** — Получение профиля (требует JWT)  
**PUT /api/user/profile** — Обновление профиля (требует JWT)

### 5.2. Сетапы

**POST /api/setups** — Создание сетапа (требует JWT)  
**GET /api/setups** — Список сетапов пользователя (требует JWT)  
**GET /api/setups/:id** — Получение сетапа (требует JWT)  
**PUT /api/setups/:id** — Обновление сетапа (требует JWT)  
**DELETE /api/setups/:id** — Удаление сетапа (мягкое, требует JWT)

### 5.3. Гонки

**POST /api/races** — Создание гонки (требует JWT)  
**GET /api/races** — Список гонок с фильтрацией (требует JWT)  
  - Query параметры: `type`, `is_completed`, `setup_id`  
**GET /api/races/:id** — Получение гонки (требует JWT)  
**PUT /api/races/:id** — Обновление гонки (требует JWT)  
**DELETE /api/races/:id** — Удаление гонки (мягкое, требует JWT)

### 5.4. Калькулятор

**POST /api/calculator/tire-pressure** — Расчёт давления (требует JWT)

**Request Body:**
```json
{
  "rider_weight": 75.5,
  "bike_weight": 8.0,
  "tire_width": 28,
  "tire_type": "tubeless",
  "surface": "asphalt",
  "conditions": "dry"
}
```

**Response:**
```json
{
  "front_pressure": 5.8,
  "rear_pressure": 6.2,
  "unit": "bar",
  "recommendations": "..."
}
```

### 5.5. Загрузка файлов

**POST /api/upload** — Загрузка фото (требует JWT)

**Request:** `multipart/form-data`, поле `file`
**Ограничения:** макс. 10 МБ, допустимые форматы: jpg, jpeg, png, gif, webp
**Response:**
```json
{
  "url": "/api/uploads/1_1773084389401602276.jpeg"
}
```

**GET /api/uploads/:filename** — Получение загруженного файла (публичный)

```
- FAB-кнопка (+) расположена над навбаром
- В standalone режиме учитывается safe-area-inset-top
- Контент не перекрывается системными элементами
```

## 8. Бизнес-логика

### 8.1. Авторизация
- Пароли хэшируются (bcrypt, cost 14)
- JWT токен: срок действия 24 часа
- Middleware проверяет токен на всех защищённых endpoints

### 8.2. Сетапы
- Пользователь видит только свои сетапы
- Давление НЕ хранится в Setup (только в Race)
- Удаление: мягкое (deleted_at)

### 8.3. Гонки
- Пользователь видит только свои гонки
- Если setup_id != null: bike_name и tires из Setup
- Если setup_id = null: bike_name и tires обязательны
- Давление всегда в Race
- is_completed = false: запланированная
- is_completed = true: прошедшая

### 8.4. Калькулятор
- Работает на основе формулы SRAM (упрощённой)
- Учитывает: вес, ширину шин, тип шин, покрытие, погоду
- Возвращает давление переднее/заднее + рекомендации

### 8.5. Уведомления
- Email "Завтра гонка": cronjob в 20:00, проверяет гонки на завтра
- Уведомление "Заполни результат": при входе, если гонка вчера и is_completed = false

### 8.6. Оффлайн режим
- Service Worker кэширует статические ассеты (stale-while-revalidate)
- API-запросы не кэшируются в SW, данные хранятся в IndexedDB
- IndexedDB: 3 хранилища — races, setups, pendingSync
- При загрузке данных с сервера — автокэш в IndexedDB
- При отсутствии сети — данные берутся из IndexedDB
- Создание/редактирование/удаление offline → записывается в очередь pendingSync
- Автосинхронизация: при появлении сети + каждые 30 секунд
- Ошибки 4xx при синхронизации — запись удаляется из очереди
- Ошибки сети — синхронизация останавливается, повтор позже
- Индикатор: жёлтый "Offline mode", синий "Syncing..." / "N pending sync"

### 8.7. Загрузка файлов
- Загрузка через multipart/form-data на /api/upload
- Максимальный размер: 10 МБ
- Допустимые форматы: jpg, jpeg, png, gif, webp
- Файлы сохраняются на сервер в директорию /app/uploads/
- Имя файла: {user_id}_{timestamp_ns}{ext}
- Файлы раздаются через /api/uploads/{filename}
- Хранилище: Docker named volume (персистентный)
- Nginx: client_max_body_size 10m

### 8.8. Калькулятор оффлайн
- Формула SRAM продублирована на фронтенде (tirePressureCalc.js)
- При ошибке вызова API — автоматический fallback на локальный расчёт
- Работает как в отдельном разделе калькулятора, так и в модальном окне из формы гонки
